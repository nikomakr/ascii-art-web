**Try to navigate between all the available pages in the website. Are all the pages working? Does the project implement 404 status?**

*How to test:*
```bash
# 200 — home page
go test -v -run "TestHomeHandler_GET" ./...

# 404 — unknown route (/about, /contact, etc.)
go test -v -run "TestHomeHandler_NotFound" ./...
```

The project has one page: `/` (home). Navigating to any undefined route (e.g. `/about`) correctly returns a `404` HTTP status. The `homeHandler` explicitly checks `r.URL.Path != "/"` and calls `renderError(w, http.StatusNotFound, ...)`. Error responses are rendered using the `error.html` template, which provides a styled HTML page with a link back to home.

**Does the project handle HTTP status 400 — Bad Request?**

*How to test:*
```bash
# 400 — invalid banner and non-ASCII input
go test -v -run "TestAsciiArtHandler_InvalidBanner|TestAsciiArtHandler_NonASCII" ./...
```

Yes. The `asciiArtHandler` and `apiAsciiArtHandler` both return `400` in three cases:
- The form/JSON body cannot be parsed
- The banner value is not one of `standard`, `shadow`, `thinkertoy`
- The input text contains a non-printable or non-ASCII character (outside 32–126, excluding `\n`)

**Does the project handle HTTP status 500 — Internal Server Errors?**

*How to test:*
```bash
# 500 — internal server error
go test -v -run "TestRenderError_InternalServerError" ./...
```

Banners are cached at startup, a missing file during runtime no longer triggers a 500 — the server would fatal at startup instead. The 500 code path (`renderError` with `http.StatusInternalServerError`) is explicitly tested in `main_test.go` via `TestRenderError_InternalServerError`.

Yes. If an internal error occurs, the handler returns a `500 Internal Server Error` response.

**Try making a request to the server (clicking a button to generate the ascii-art representation on the website). Is the communication between server and client well established?**

*How to test:*
```bash
# 200 — valid request to generate ASCII art
go test -v -run "TestHomeHandler_GET|TestAsciiArtHandler_ValidInput" ./...
```

Yes. The frontend uses the browser `fetch()` API to send a `POST` request with JSON `{"text","banner"}` to `/api/ascii-art`. The server processes the request and returns a JSON response `{"result":"..."}` — the result is displayed on the page without a full reload, confirming the communication is correctly established.

---

**Does the server use the right HTTP method?**

Yes. Each endpoint uses the semantically correct HTTP method:

| Endpoint         | Method | Reason                                      |
|------------------|--------|---------------------------------------------|
| `/`              | GET    | Retrieves and renders the main page         |
| `/ascii-art`     | POST   | Submits form data to generate ASCII art     |
| `/api/ascii-art` | POST   | Submits JSON data to the REST API           |

All handlers explicitly validate the incoming method and return `405 Method Not Allowed` if an incorrect method is used.

*How to test:*
```bash
# 405 — wrong HTTP methods
go test -v -run "TestHomeHandler_MethodNotAllowed|TestAsciiArtHandler_MethodNotAllowed" ./...
```

**Did the site work without crashing at any time?**

Yes. The server remained stable throughout all testing scenarios, including:

- Valid and invalid text inputs
- All three banner styles
- Incorrect HTTP methods (405)
- Unknown routes (404)
- Malformed or missing banner files (500)

Go's `net/http` package isolates each request in its own goroutine, so a single failing request cannot bring down the server. All error paths return a proper HTTP response rather than panicking, ensuring continuous uptime.

## General

**Does the server present all the needed handlers and patterns for the HTTP requests?**

Yes. The server registers three handlers covering all required routes:

| Handler               | Route            | Method |
|-----------------------|------------------|--------|
| `homeHandler`         | `/`              | GET    |
| `asciiArtHandler`     | `/ascii-art`     | POST   |
| `apiAsciiArtHandler`  | `/api/ascii-art` | POST   |

All invalid methods return `405 Method Not Allowed` and all unregistered routes return `404 Not Found`.

*How to test:*
```bash
# 200 — registered routes respond correctly
go test -v -run "TestHomeHandler_GET|TestAsciiArtHandler_ValidInput" ./...

# 404 — unregistered routes
go test -v -run "TestHomeHandler_NotFound" ./...

# 405 — wrong methods on registered routes
go test -v -run "TestHomeHandler_MethodNotAllowed|TestAsciiArtHandler_MethodNotAllowed" ./...
```

## Basic

**Does the server run quickly and effectively?**

*How to test:*
```bash
# Benchmark home handler and ASCII art handler
go test -v -bench=. -benchtime=5s ./...
```

Template parsing happens once at startup using `template.ParseGlob`. Banner files are also loaded once at startup into `bannerCache` and reused on every request — avoiding any disk I/O in the hot path. There are no unnecessary loops or redundant allocations per request.

**Does the code obey good practices?**
Yes. The code is well-structured with clear separation between the HTTP layer (`homeHandler`, `asciiArtHandler`, `apiAsciiArtHandler`) and the logic layer (`generateASCIIArt`, `loadBanner`). Input validation is thorough. Banner files are loaded once at startup into `bannerCache` and reused across all requests, avoiding repeated disk reads. Templates are parsed once at startup via `template.ParseGlob`.

**Is there a test file for this code?**
Yes. `main_test.go` covers all HTTP handlers and core logic functions.

*How to test:*
```bash
# Run all tests and list each one by name
go test -v ./...
```

**Are the tests checking each possible case?**
Yes. Tests cover:
- `TestHomeHandler_GET` — home page returns 200 OK
- `TestHomeHandler_NotFound` — unknown path returns 404 Not Found
- `TestHomeHandler_MethodNotAllowed` — wrong method on `/` returns 405 Method Not Allowed
- `TestAsciiArtHandler_ValidInput` — valid form POST returns 200 OK
- `TestAsciiArtHandler_InvalidBanner` — invalid banner returns 400 Bad Request
- `TestAsciiArtHandler_NonASCII` — non-ASCII input returns 400 Bad Request
- `TestAsciiArtHandler_MethodNotAllowed` — wrong method on `/ascii-art` returns 405 Method Not Allowed
- `TestRenderError_InternalServerError` — `renderError` correctly writes 500 Internal Server Error
- `TestGenerateASCIIArt_EmptyText` — empty input returns empty result
- `TestGenerateASCIIArt_Newline` — newline input is preserved in output

The test `init()` initialises both `templates` and `bannerCache` so all tests run with the full server state.

**Are the instructions in the website clear?**
The interface is minimal but functional. Labels clearly indicate "Enter your text" and "Choose a banner". The result is shown under a "Result:" heading. Additional guidance (e.g. character restrictions, multiline support) could improve the user experience.

**Does the project run using an API?**
Yes. The project exposes a REST/JSON API endpoint at `POST /api/ascii-art`. The frontend uses the browser `fetch()` API to send a JSON request `{"text": "...", "banner": "..."}` and receives a JSON response `{"result": "..."}` — no full page reload occurs. The traditional form handler at `POST /ascii-art` is still present for fallback, but the primary client–server communication goes through the JSON API.
